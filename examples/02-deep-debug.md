# 例 02 · 沿调用链定位「窾」(Deep Debug)

锁定本 skill 在「深层 bug、无从下手」场景下,如何用「找缝隙」而非「乱改」定位根因。

---

## INPUT(用户请求)

> 下单接口偶发返回 200 但订单没落库,日志没报错。改了三处都没用。怎么办?

## 没有本 skill 的产出(反模式)

```
我在 createOrder 里加个 try/catch 吧……
再加个重试……再把事务超时调大……
（在耦合最深、副作用最多的 createOrder 主干上连续乱改,
 每改一处牵动半个下单流程,无法判断哪一改有效）
```

## 有本 skill 的产出(循纹理定窾)

```
[庖丁 | 目标:偶发 200-但-不落库 | 第一刀:沿调用链找「吞掉错误的空处」(窾)| 硬骨头:分布式事务边界(軱)| 余地:加观测不改行为,先复现再下刀]
```

### 1 观(依乎天理)— 先画这条牛的筋络
不立刻改。先沿调用链建模型:

```
createOrder → validate → tx.begin → insertOrder → publishEvent → tx.commit → 200
```

逐节点问:**哪一节「有间」(本可静默失败的空处)?**

### 2 寻(批大郤·导大窾)— 找吞错误的缝隙
偶发 + 200 + 无错误日志 = 典型「错误被吞在某个空处」。
循纹理排查接缝,不碰主干:
- `publishEvent` 是否在 `tx.commit` **之前**且其失败被 catch 吞掉?→ 这就是窾。
- 复现:对该节点注入失败,观察是否复现「200 但不落库」。

### 3 慎(怵然为戒)— 事务边界是軱,降速
`tx.begin/commit` 跨服务——分布式事务边界是硬骨头。
- **不**直接调超时/加重试(那是在軱上硬砍)。
- 先加结构化观测(span/日志)定位提交是否真正发生,再下刀。

### 4 养(刀刃若新)— 小步、可逆
1. `obs: createOrder 全链路加 span,定位静默失败点`(纯加观测,零行为改动,可回退)
2. 确认根因后,单一最小修复:把 `publishEvent` 移到 commit 之后 / 改为出箱表。
3. 补一条复现该 race 的回归测试。

---

**要点**:深层 bug 的「根因」往往藏在某个「空处」(被吞的错误、事务边界外的副作用)。
庖丁之刀走空处——先沿调用链找那道缝,而不是在主干上反复硬砍。
