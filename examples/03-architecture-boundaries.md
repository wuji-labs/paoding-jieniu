# 例 03 · 划分系统边界(Architecture Boundaries)

锁定本 skill 在「从零设计 / 划分模块边界」场景下,如何用「天理」找自然结构线。

---

## INPUT(用户请求)

> 我们要做一个「会员积分 + 优惠券 + 等级权益」系统。该怎么划模块?别让它最后耦成一坨。

## 没有本 skill 的产出(反模式)

```
建一个 MembershipService,把积分、优惠券、等级都放进去……
再共用一张大表 user_benefits 存所有状态……
（按「功能堆在一起」划,无视领域的自然纹理,半年后又耦成一坨)
```

## 有本 skill 的产出(依乎天理找边界)

```
[庖丁 | 目标:积分/券/等级 模块划分 | 第一刀:沿领域自然纹理切三块(郤)| 硬骨头:跨域一致性(等级随积分变)(軱)| 余地:事件解耦,先各自独立可测]
```

### 1 观(依乎天理)— 找领域本有的结构线
设计前先问:这三者**本来**就是几头牛?各自的不变量(invariant)是什么?

| 子域 | 核心不变量 | 自然纹理 |
|------|-----------|---------|
| 积分 Points | 余额 ≥ 0、流水可追溯 | 一本账(账户 + 流水)|
| 优惠券 Coupon | 有效期、单券单用、可核销 | 状态机(发放→可用→核销/过期)|
| 等级 Tier | 由累计积分推导 | 派生视图(读模型)|

天理:**积分是账本,券是状态机,等级是派生**——三者结构形态本就不同,不该塞一坨。

### 2 寻(批大郤·导大窾)— 沿郤切三块
按这条纹理切出三个边界清晰的模块,各自拥有自己的数据:
- `points/`(账户 + 流水,只读 API 对外)
- `coupon/`(券状态机,核销是其唯一写入口)
- `tier/`(订阅积分变化事件,派生等级,**不拥有写状态**)

### 3 慎(怵然为戒)— 跨域一致性是軱
「积分到阈值自动升等级」是跨域不变量——硬骨头。
- **不**用同步跨表写(那是把三头牛缝回一坨)。
- 用 `PointsAccrued` 领域事件让 tier 异步派生;降速,先把一致性语义(最终一致 vs 强一致)写清再下刀。

### 4 养(刀刃若新)— 各自可独立测
- 三模块各带独立测试,事件契约用 schema 锁定。
- 先交付 points + coupon 各自闭环,tier 派生后接——小步、可逆,不一次性大耦合上线。

---

**要点**:好的边界不是「设计」出来的,是「读」出来的——
顺领域本有的不变量(账本/状态机/派生)切,就是依乎天理;逆着塞一坨,半年必折刀。
