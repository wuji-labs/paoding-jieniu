# PaoDing JieNiu — Benchmark Pilot Run (2026-06-14)

> **This is a real run, honestly reported.** It replaces the previous
> "RESULTS NOT YET RUN" placeholder for the two scenarios below. No number here is
> fabricated; the raw transcripts are included verbatim so the finding is
> independently checkable.

## Method (what was actually done)

- **Harness:** the agent-in-the-loop variant of `run_benchmark.py`'s design — each
  trial is a fresh agent given the scenario `task` + read access to the real
  `test-project/tangled-orders/` files, with ground truth withheld.
- **Conditions:** `baseline` (generic senior-engineer framing, skill not read) vs
  `paoding` (the agent first reads and applies `SKILL.md`). Same task, same code,
  only the methodology differs.
- **Model:** Claude Opus 4.x (the agent runtime of this environment). **No pinned
  API model** — this is the honest limitation: results characterize a *frontier*
  model, not a weak one.
- **Scenarios:** #1 (decomposition — extract invoice from god object) and #2
  (debugging — 200 OK but order not persisted). 3 runs per condition per scenario
  → **N = 12 transcripts.**
- **Scoring:** `expected_actions` is an objective checklist (did the answer do each
  listed thing?). The 0–3 dimensional rubric is subjective and is **not** scored
  here; doing it credibly needs a blind judge and is listed as next-step.

## Result

| Scenario | Condition | Runs | expected_actions hit-rate | Correct root cause / first cut |
|---|---|---|---|---|
| #1 decomposition | baseline | 3 | **5/5 (100%)** all runs | 3/3 — `invoice/render.py` first |
| #1 decomposition | paoding | 3 | **5/5 (100%)** all runs | 3/3 — `invoice/render.py` first |
| #2 debugging | baseline | 3 | **5/5 (100%)** all runs | 3/3 — publish-before-commit + swallow |
| #2 debugging | paoding | 3 | **5/5 (100%)** all runs | 3/3 — publish-before-commit + swallow |

**No measurable gap between conditions.** On both scenarios, every baseline run
independently did the structure-first thing the skill prescribes: read the
dependency direction before editing, picked the pure seam (`invoice/render.py`) as
the first cut, refused to touch the god object first, demanded golden tests before
moving the `Money`-coupled `calc.py`, walked the call chain instead of hacking the
trunk, and found the swallowed pre-commit `publish`. Several baseline runs were, if
anything, *more* careful (two explicitly noted that control still falls through to
`commit` in the exact seeded code — a subtlety).

## Honest interpretation

This is the expected and correct result, not a disappointment:

1. **A frontier model is at ceiling on a 150-LOC tangle.** When the base model
   already reasons structurally, a structure-first *prompt* has little headroom to
   add. The skill's measured lift here is ≈ 0 because the baseline is ≈ 100%.
2. **Where the skill is expected to pay off** — and what this pilot does **not**
   yet measure — is: (a) **weaker / smaller models** that don't structure-decompose
   by default; (b) **larger, messier real codebases** where the seam isn't obvious
   in one read; (c) **consistency / floor-raising** across many runs and many users,
   not peak capability on an easy case.
3. Measuring (a) — weaker models — has since been **done** via local Ollama (no API
   key, zero cost): see [`weak-model/FINDINGS.md`](./weak-model/FINDINGS.md). The
   honest result there is **unfavorable**: on a 7B model the heavyweight skill
   slightly *hurt* (correct 72% vs baseline 100%), because prepending a long
   methodology crowds a small model's attention. That validates a lean variant and
   shows our small inlined subject can't probe the skill's real target
   (large-codebase navigation). We report it as measured.

**Bottom line:** the skill does not *hurt* a frontier model (it matches baseline at
ceiling) and encodes the exact method a strong model already uses — its value
proposition is reliability and weaker-model lift, which remain to be quantified.
We report what we measured and label clearly what we did not.
